In web terms, this is what Client-Side Rendering does. Basically, when you visit a website that uses CSR, your web browser is like the waiter. It receives a basic set of instructions and ingredients (HTML and JavaScript), and it assembles the webpage on your device. This allows for dynamic and interactive experiences, like animations and real-time updates, but it might take a little longer for the page to show up initially because your browser has to do some work to put everything together.

Server Side Rendering (SSR):

Now, let's go back to the restaurant, but this time it's a bit different. You order the dessert, and the waiter goes to the kitchen to prepare it for you. They bring the finished dessert to your table, and you can start savoring it immediately.
In web terms, Server-Side Rendering works similarly. When you visit a website that uses SSR, the server (like the kitchen) prepares the complete webpage in advance, including all the toppings and decorations. Then, it sends the fully baked webpage to your browser. As a result, the page appears faster because most of the work has already been done on the server.

Cool features of SSR
#1. Faster Websites: SSR makes websites open up faster because a lot of the work is already done before the site gets to you. This is especially helpful if you're using a slow internet connection or a not-so-powerful device.

#2. Better Search Results: Search engines like Google prefer websites that use SSR. This makes it easier for them to understand what a website is about and show it to people who are looking for similar things.

#3. Good for Sharing: When you share a link on social media, SSR helps show a preview of the website's content. This is important because you want the preview to match what's on the actual page.

#4. Works When Stuff Breaks: Sometimes, parts of a website might not work properly because of technical issues. SSR helps by making sure you can still see the main stuff even if the fancy parts aren't working.

And there are a lot more which you can find in the internet...


Then why not we "just" use SSR
with great power comes great responsibilities, and SSR sometimes fails to handle some of those responsibilities

Basically,
#1. Slower Initial Load on Server: SSR requires the server to do more work before showing you a website. This can sometimes slow down the server and make it take longer to send you the page.

#2. More Server Resources: Because SSR does extra work on the server, it needs more power and resources. This can be a problem if a lot of people are trying to use the website at the same time.

#4. Limited Client-Side Interactivity: While SSR makes things load faster, it can limit some of the fancy interactive features that make websites really cool. Some of these features might not work as smoothly with SSR.

#5. Caching Challenges: Caching, which helps store and reuse information to speed up websites, can be a bit tricky with SSR. It requires careful handling to make sure the cached content is always up-to-date.

Conclusion üé∫

Like every other thing in life, SSR has its advantages and disadvantages. When to use CSR and when to use SSR is very highly dependent on the type of website you're trying to build.

My opinion, if your website will require a lot of user interaction and dynamic updates (like real-time chats or interactive maps), CSR might be a better fit. However, if search engine optimization (SEO) and social sharing are important for your application, go with SSR.
Everything else seems managable.
